접근성이라고는 찾아볼 수 없는 html 코드

위 드롭다운 내부의 코드를 보게되면, table, tr, td 만으로 테이블을 표현하고 있다. 사실 이렇게만 표현해도 자료를 충분히 표현할 수 있고, css를 이용해서 충분히 예쁘게 꾸밀 수도 있을 것이다. 하지만 전혀 sementic 한 느낌을 주지도 않고 접근성을 아예 고려하지 않은 코드이다.

그럼 어떻게 수정할 수 있을까?

2. 접근성을 고려한 table 태그 활용하기

table과 관련한 태그들과 속성들을 먼저 알아보겠다.

  • tags

    • thead, tbody, tfoot 3형제
    • th, tr, td
    • caption
    • colgroup, rowgroup, col, row

  • attributes

    • scope, id, headers
    • summary (Deprecated)

사실 스스로도 검색하고 공부해가면서 정리하기 전까지 표를 그리기 위한 간단한 table 태그에 이렇게 많은 속성들과 관련 태그들이 존재하는 줄 모르고 있었다. 이렇게까지 많은 태그와 속성들의 필요성을 모르고 있었기 때문인 것 같다.

위에서 예시로 든 표처럼 행과 열이 명확하게 구분되는 간단한 표의 경우에는 scope에 row와 col을 주면 된다는 것을 이해했지만, 만약 셀들이 병합되고, 헤더들이 병합되어 의미하는 주제들이 소주제로 쪼개진다면 어떻게 처리를 해야할지 몰랐다. 하지만 아래 W3C - Table with irregular headers 에는 아래와 같이 설명이 되어있는 것을 보고 어떻게 처리해주어야 할지 이해할 수 있었다.

즉, Mars 와 Venus 로 구분된 열 하위에 Produced, Sold라는 구분이 각각 되어있는 것을 확인할 수 있는데 이렇게 열의 헤더가 단순히 하나만으로 구성되어 있지 않은 경우에는 colgroup 속성을 사용해서 명시하라고 되어있는 것을 확인할 수 있다. 그리고 첫 번째 레벨의 헤더에 각각 scope 속성을 주어 해당 열이 연관되어 있다는 사실을 명시하라고 되어있다.

그리고 위 그림(표)는 w3c에서 WAI-ARIA와 TABLE에 관련한 정보 페이지이다. 이것을 보게되면 aria의 role로 table, rowgroup, row, columheader, cell 등이 존재하고, attribute로는 aria-label로 테이블의 값을 주고 aria-describedby로 테이블의 정보(caption)을 간단히 명시해주는 것을 볼 수 있다. WAI-ARIA에 대해서 모르고 있었던 부분이 너무 많았다는 것을 느꼈고, 간단한 table 태그에 조차 이렇게 많은 접근성을 고려할 수 있다는 것을 깨달았다.

위 코드는 WAI-ARIA를 적용한 예시 코드 중 일부이다. 그러니까 table 태그나 tr, td와 같은 semantic 하지 않은 태그들을 사용했을 경우에 부수적으로 기재해주는 속성인 것이다. 아무래도 semantic 한 태그들과 속성들을 사용해주는 것이 더 좋아보이기 때문에 뒤의 실습에서는 table, thead, tbody, tr 등의 태그를 사용하고 caption, scope, id, headers를 활용해 테이블을 그려보았다.

아래 있는 실습 단계에서 이를 참고하여 접근성이 높은 테이블을 구성해보았기 때문에 나중에 확인해볼 수 있고, 여기에서는 접근성에 매우 중요한 몇 가지 중요한 정보들에 대해 자세히 알아보겠다.

2.1. caption

일단 caption 태그는 해당 표에 대한 설명, 목적 등을 알려주는 태그로, 사용자가 표 컨텐츠를 미리보기처럼 빠르게 확인하여 자신에게 필요한 정보인지 아닌지 판단할 수 있도록 해주는 접근성의 기본 중의 기본이 되는 태그이다. 일반적인 사람들에게도 충분히 도움이 되겠지만 특히 스크린 리더와 같은 보조 기술 사용자나, 시력관련 장애인들에게 큰 도움이 된다.

2.2. thead, tbody, tfoot 3형제

이 3가지 태그들은 테이블의 행을 그룹짓는 semantic 한 태그들이다. 이러한 태그들을 활용해 위의 caption태그와 같이 미리 정보들을 유추할 수 있기 때문에 접근성에 아주 중요한 역할을 한다고 볼 수 있다.

2.3 scope, id, headers

시력에 문제가 없는 일반적인 사람들의 경우에는 테이블을 한 눈에 파악할 수 있고, 어떤 주제에 대한 표인지 어떤 행이 어떤 의미를 가지고 어떤 열이 어떠한 의미를 가지는지 곧바로 이해할 수 있다. 하지만, 시력에 문제가 있는 일부 사용자의 경우에는 보조 장치 등을 이용하여 이를 파악해야 하는데 테이블이 복잡해진다면 각각의 자료들(셀)이 어떠한 행과 어떠한 열과 연관성이 있는지 알기가 쉽지 않다.

따라서 위 3가지 속성들을 이용하여 이를 구분해주는 과정을 통해 접근성을 향상시킬 수 있다. 예를 들어서, th 요소에 scope 속성을 지정하고 해당 값으로 col이나 row, rowgroup, colgroup를 할당해 해당 셀이 열의 제목인지 행의 제목인지 등을 알 수 있다. 만약 더 복잡한 경우, headers 속성과 id 속성을 이용해 내용과 그 의미를 연관지을 수도 있다.

따라서, 아래와 같이 수정한다면 접근성을 조금 더 향상 시킬 수 있을 것이다.

그리고 만약 셀들이 병합되는 등 좀 더 복잡한 테이블의 경우에는 colgroup과 rowgroup속성을 통해 col, row 등을 그룹지어주는 과정도 필요하다.

3. 실습

위 그림은 간단한 표 예제이다. 사실 만약 내가 접근성이 뭔지도 몰랐을 때 위 표를 마크업한다고 하면, 아마도 다믕과 같은 형식으로 짰을 것이 분명하다.

일단 위 코드를 보면

  • caption 태그가 있는가? “NO”
  • thead, tbody, tfoot이 있는가? “NO”
  • scope속성을 주었는가? “NO”

접근성에 대해 단 ‘1’도 고려하지 않은 코드라고 볼 수 있겠다. 하지만 과거의 나라면 단순히 표가 잘 그려지는 것을 보고 만족했을 것이다.
하지만, 접근성을 조금 더 고려해본다면 다음과 같이 개선할 수 있을 것이다.

  • caption 태그가 있는가? “YES”
  • thead, tbody, tfoot이 있는가? “YES”
  • scope속성을 주었는가? “YES”

이렇게 수정함으로써 디자인적으로는 변함없이, 스크린 리더가 표를 효과적으로 읽어주어 시각적으로 불편한 사람들이 보조기기의 도움을 받아 표를 조금 더 잘 이해할 수 있도록 해줄 수 있게 되었다.

그리고 복잡한 테이블의 경우에는 scope 속성을 사용하지 않고 id, headers를 사용해 조금 더 명확히 각 셀의 내용을 전달할 수 있다. 예를 들어서 아래와 같은 표를 표시해야 한다고 하자.

이 경우에, 표의 헤더의 각각의 구분 단위들에 따른 각 셀들의 연관성이 조금 비교적 복잡하게 나타나있고 시각적으로 이를 확인할 수 없는 상황이라면 표의 내용을 이해하는데 어려움이 있을 것이다. 따라서 아래 코드와 같이 id, headers를 주어 각 셀들이 가지는 연관성과 의미를 조금 더 명확히 해줄 수 있다.

descender, ascender

HTML

1993년 최초버전. W3C가 XHTML1.0을 구체화하기 위해 XHTML2.0작업을 진행을 시도

  • HTML 4.01 vs XHTML 1.0

    • 재정의 한 것이기 때문에 비슷함
    • HTML 은 자유도가 너무 높았음
    • 애플, 모질라, 오페라가 w3c와 별개로 작업함(WHATWG)

  • HTML5 정보 구조를 명확히 할 수 있도록 아웃라인 알고리즘(?)이라는 것이 있음

  • HTML5는 HTML 4.01과 XHTML1.0을 모두 수용했다.

Naming

  2. K (kebob): main-content
  3. C (camel): mainContent

    • P: Property

HTML 표 컨텐츠와 양식

표 컨텐츠

table, tr, th, td

  • table: display=table
  • tr: display=table-row
  • th,td: display=table-cell
  • th는 테이블의 헤더 역할을 한다(글씨가 두껍게 되고 등등)

th

  • abbr : 열에 대한 간단한 소개
  • headers: 관련된 하나 이상의 다른 머리글 칸 id속성 값
  • colspan
  • rowspan
  • span: 자신이 누구 머리글 칸인지 명시 (col, colgroup, row, rowgroup, auto(row))

td

  • headers : 보통 th의 id에 서연결한다.
  • colspan
  • rowspan

caption

  • display는 table-caption이다
  • 테이블 태그 바로 하위에 작성하며 테이블 제목의 역할을 수행한다

양식 태그

<form>

웹 서버에 정보를 제출하기위한 양식의 범위를 정함

  • action: 전송한 정보를 처리할 url
  • autocomplete: 사용자가 이전에 입력한 값으로 자동완성 할지 여부
  • method: 서버로 전송할 방식
  • novalidate: 유효성을 검사하지 않도록 지정
  • target: 서버로 전송 후 응답받을 방식을 지정

<input>

사용자에게 입력받을 데이터 양식

  • autofocus: 페이지가 로드될 때 자동으로 포커스

  • form: 폼 바깥에 작성한 것을 이어주는 역할

  • type: 입력할 수 있는 종류

    • button
    • checkbox
    • color 등
    • 예를 들어서,type=number과 같이 숫자를 입력하려고 했으면 숫자만 입력이 됨
    • image 이미지로 끝나지 않고, 제출 버튼으로 활용할 수 있다.
    • reset 데이터는 자신이 포함된 form태그 내의 input들을 초기화한다.

  • placeholder: 사용자에게 힌트를 줌

<label>

라벨 가능 요소(labelable)의 제목(Caption).

  • for 속성으로 라벨 가능 요소를 참조하거나 콘텐츠로 포함.
  • 라벨 가능 요소: <button>, <input>, <progress>, <select>, <textarea>

<button>

  • autofocus 페이지가 로드될 때 자동으로 포커스
  • disabled 버튼을 비활성화
  • form <form>의 id 속성 값
  • name 폼 데이터와 함께 전송되는 버튼의 이름
  • type 버튼의 타입
    • button, submit, reset

<textarea>

여러 줄의 일반 텍스트 양식

  • display: inline-block
  • rows 양식의 줄 수 숫자(Number)
    • default: 2

<fieldset>, <legend>

같은 목적의 양식을 그룹화(<fieldset>)하여 제목(<legend>)을 지정

멀티미디어 & 내장 콘텐츠

audio Tag

소리 콘텐츠(MP3)를 삽입, inline 요소

  • autoplay 준비되면 바로 재생 불린(Boolean)
  • controls 제어 메뉴를 표시 불린(Boolean)
  • loop 재생이 끝나면 다시 처음부터 재생 불린(Boolean)
  • preload 페이지가 로드될 때 파일을 로드할지의 지정(힌트 제공)
    • none: 로드하지 않음
    • metadata: 메타데이터만 로드

  • src 콘텐츠 URL
  • muted 음소거 여부 불린(Boolean)

video Tag

동영상 콘텐츠 삽입, inline 요소

  • autoplay 준비되면 바로 재생 불린(Boolean)
  • controls 제어 메뉴를 표시 불린(Boolean)
  • loop 재생이 끝나면 다시 처음부터 재생 불린(Boolean)
  • muted 음소거 여부 불린(Boolean)
  • poster 동영상 썸네일 이미지 URL
  • preload 페이지가 로드될 때 파일을 로드할지의 지정(힌트 제공)
    • none: 로드하지 않음
    • metadata: 메타데이터만 로드 (기본값)
    • auto: 전체 파일 로드
  • src 콘텐츠 URL URL
  • width 동영상 가로 너비
  • height 동영상 세로 너비

figure, figcaption tag

iframe Tag

다른 페이지를 현재 페이지에 삽입(HTML)

  • inline 요소
  • iframe tag(html)에다가 height, width 를 정할 수 있음

iframe 속성들

  • name
  • frameborder : 기본 값은 1
  • allowfullscreen
  • sandbox: 보안을 위한 읽기 전용으로 삽입s
    • allow-form
    • allow-scripts
    • allow-same-origin: 같은 출처(도메인) 리소스 사용 가능

멀티미디어 & 내장 콘텐츠 Tag

<img> Tag

이미지 삽입을 위한 태그

  • src: (필수)이미지 URL
  • alt: (필수)이미지의 대체텍스트
  • width: 이미지의 가로 너비 (px 없이 숫자만)
  • height: 이미지의 세로 너비 (px 없이 숫자만)
  • srcset: 브라우저에게 제시할 이미지 URL과 원본 크기의 목록을 정의
  • sizes: 미디어 조건과 해당 조건일 때 이미지 최적화 크기의 목록을 정의

srcset

srcset & sizes link

브라우저에 제시할(사용할) 이미지들과 그 이미지들의 원본 크기를 지정

  • 사이즈별로 2장 이상 준비하여 srcset 속성에 작성
  • px단위가 아닌 w 디스크립터 혹은 x 디스크립터를 입력해야 하며, 작은 크기 이미지부터 순서대로 입력
  • 늘려서 사용하는 것 보다는 줄여서 사용할 수 있는 가장 가까운 크기의 사진을 선택

W unit

Width descriptor)는 이미지의 원본 크기(가로 너비)를 의미

  • 400x300(px) 크기 이미지의 w 값은 400w
  • 고정된 이미지 크기를 유지하려면 width 속성을 추가
    sizes 속성과는 다른 개념

X unit

x 디스크립터(Device pixel ratio descriptor)는 이미지의 비율 의도

  • 디바이스의 픽셀 비율(Device pixel ratio)과 일치하는 값으로 최적화 선택
  • 정수(integer) 값으로 제공하는 것이 좋음

2

sizes

미디어조건과 그 조건에 해당하는 이미지의 ‘최적화 출력 크기’ 를 지정

결론

srcset & sizes link

  • width는 이미지의 ‘출력 크기’만 지정
  • sizes는 이미지의 ‘출력 크기’ + ‘최적 크기’도 함께 지정
  • 따라서 sizes=”500px”이 지정된 첫번째 예제는 500px에 최적화된 이미지로 heropy_medium.png를 사용했고 이미지 크기도 500px로 설정
  • sizes와 width를 같이 작성할 경우 width가 우선
  • srcset과 sizes는 IE를 지원하지 않음

인라인 텍스트

A Tag

다른 페이지, 같은 페이지 위치(#, 해시 태그), 파일, 이메일 주소, 전화번호 등 다른 URL로 넘어가는 하이퍼링크, Anchor

속성들

  • download: 리소스를 다운로드하는 용도로 사용됨을 의미 (Boolean)
  • href: 링크 URL
  • hreflang: 링크 URL 언어 (ko, en 등)
  • rel: 현재 문서와 URL강늬 관계 (license, prev, next 등)
  • target: 링크 URL 위치 (_self, _blank)
  • type: 해당 URL의 MIME 타입 (test/html)

Span Tag

본질적으로 아무것도 나타내지 않은 콘텐츠영역을 설정

HTML 콘텐츠 구분

BODY 내 구조 개념

body 태그는 정보의 구조들을 나타냄

  • 의미가 없는 div 태그들로 도배하면 좋지 않다.
  • header 태그, ul, li 등 의미를 가지는 태그들을 사용해야 한다.
  • 일반적으로 영역을 나타내는 태그들은 block요소이다

콘텐츠 구분

Header Tag

소개나 탐색을 돕는 그룹을 나타냄

  • 로고
  • 검색바
  • 메뉴 등

header나 footer는 후손이 될 수 없다.

Footer Tag

작성자 구획, 저작권 데이터, 관련 문서 등에 대한 링크, 정보를 포함한다.
일반적으로 사이트 하단의 정보들을 포함

header나 footer는 후손이 될 수 없다.

H Tag

6단계의 문서 제목을 지칭
h1 ~ h6까지 존재

  • 크기 조절을 위해서 h 태그를 사용하는 것은 좋지 않고, 대신 css의 font-size 속성을 사용해야 함
  • 제목 단계를 건너뛰면 좋지 않음
  • h1은 한 페이지에 하나씩만 사용해야 함
  • 추천하는 방식정도

Main Tag

<body> 의 주요 콘텐츠를 나타내는 태그
문서 핵심 주제 및 핵심 기능성 등과 직접적으로 연관된 콘텐츠들로 구성

  • 문서에 하나의 <main>만 존재해야 함
  • IE에서 지원하지 않음

Article Tag

독립적으로 구분되거나 재사용 가능한 영역을 설정
매거진/신문 기사, 블로그 등

  • 일반적으로 <h1> ~ <h6>를 포함하여 식별.
  • 작성일자와 시간을 <time>의 datetime 속성으로 작성.
  • article 태그 안에 section 태그가 있어도 되고, 그 반대도 가능함

Section Tag

문서의 일반적인 영역을 설정

  • 영역을 의미하기 때문에 일반적으로 <h1> ~ <h6>를 포함하여 식별
  • div와 차이: 의미가 있기 때문에 h태그로 제목을 포함함
  • 독림성을 가지는 article 태그와 함께 사용할 수 있음

Aside Tag

문서의 별도 콘텐츠를 설정

  • 보통 광고나 기타 링크 등의 사이드바를 설정

Nav Tag

다른 페이지 링크를 제공하는 영역

  • Navigation, 보통 메뉴(Home, About, Contact), 목차, 색인
  • ol, ul 태그와 함께 메뉴들을 보여줄 때 자주 사용됨

Address Tag

<body>, <article>, <footer> 등에서 연락처 정보를 제공하기 위해 포함

  • block 요소

Div Tag

본질적으로 아무것도 나타내지 않는 콘텐츠 영역

  • Division, 꾸미는 용도

문자 구분

ol, ul, li Tag

각 항목<li> 의 정렬된 목록<ol>이나 정렬되지 않은 목록<ul>을 설정

  • <ol>과 <ul>은 자식으로 <li>만 포함 가능
  • <li>는 단독으로 사용할 수 없으며, <ol>이나 <ul>에 자식으로 포함되어야 함
  • 정렬된 목록(<ol>)의 항목 순서는 중요도를 의미할 수 있음

ol 속성

  • start: 항목에 매겨지는 번호의 시작 값 숫자(Number)
  • type: 항목에 매겨지는 번호의 유형 a, A, i, I, 1
  • reversed: 항목의 순서만 바뀌고 물리적 위치는 바뀌지 않음

li 속성

  • value: 항목의 순서를 설정 숫자(Number) 이하 항목들의 순서가 다시 지정됨

dl, dt, dd

용어(<dt>)와 정의(<dd>) 쌍들의 영역(<dl>)을 설정

  • block 요소
  • Description List, Definition Details, Definition Term
  • <dl>은 <dd>, <dt>만을 포함해야 함
  • 키(key)/값(value) 형태를 표시할 때 유용
  • 근데 dl 태그안에 div 와 같은 태그를 넣을 수 없기 때문에 스타일링하기 힘듬
