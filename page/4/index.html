제어문

제어문은 어떠한 프로그래밍 언어든 아주 중요한 역할을 한다. 주어진 조건에 따라서 특정한 코드 블록을 실행하거나 반복할 때 사용하는데, 이렇게 제어를 통해 코드의 실행 흐름을 제어할 수 있다. 하지만 흐름을 제어한다고해서 다 좋은 것은 아니고 코드의 흐름이 복잡해져 가독성 측면에서 좋지 않을 수도 있기 때문에 제어문을 남발하면 안된다.

반복문

반복문도 프로그래밍의 흐름 제어에서 아주 중요한 역할을 하고 시간 복잡도에 영향을 주는 메인 요소이다. 조건에 따라 일정한 코드 블록을 반복적으로 수행하는 것이고, 이 때 조건을 잘 체크하여 무한루프가 형성되지 않도록 하는 주의가 필요하다.

자바스크립트에서는 기본적으로 for, while, do while 문을 제공한다.

자바스크립트는 forEach, map, filter 등 다양한 메서드들이 존재하며 반복문처럼 Array의 각 요소를 iterate 할 때 사용된다.

break

이 때 while 문에서 조건식에 true 가 되는 값을 주어 무한루프를 만든 후, 반복문 내부에서 조건식을 만족했을 때만 루프가 종료되도록 if 조건문으로 break 키워드를 사용해 반복문을 빠져나올 수 있다.

데이터 타입

데이터 타입이란 값의 “종류”를 말하는 것이다. 123 은 숫자 타입이고(참로고 자바스크립트에는 정수형이나 실수형이 구분되지 않고 모두 실수로 처리된다), hello 는 문자 타입이다.

원시 타입(primitive type)

자바스크립트에는 총 6개의 원시 타입이 존재한다

  • 숫자 타입 (number): 정수와 실수를 구분하지 않고 숫자 타입으로 존재

  • 문자열 타입 (string): 문자열

  • 불리언 타입 (boolean): 논리적 참과 거짓 (true/false)

  • undefined 타입: undefined var 키워드로 선언된 변수에 암묵적으로 할당되는 값

  • null 타입: 값이 없다는 것을 의도적으로 명시하는 값

  • 심볼(symbol): ES6 추가된 타입

그리고 객체 타입(object/reference type)

헷갈리지 않도록 하기 위해 예시를 하나 들겠다.

위에서 one은 숫자 타입, two는 문자열 타입이다.

각 타입별 특징

Number 타입

숫자 타입엔 특별한 세 가지 값들이 존재한다.

  • Infinity
  • -Infinity
  • NaN

자바스크립트는 대소문자를 구분하기 때문에 nan 과 같이 대소문자를 무시하고 사용할 수 없다는 점을 주의해야 한다.

String 타입

일반적으로 " ... " ' ... ' 와 같이 큰 따옴표와 작은 따옴표로 감싸는 방법 외에도 백틱(`) 으로도 감쌀 수 있는데, 문자열 표현식을 삽입하는 경우에 매우 편리하고 가독성이 좋다.

위와 같이 사용할 수 있으며 가독성이 좋은 것을 확인할 수 있다.

undefined 타입

자바스크립트를 배우며 처음에 undefined 타입과 null 을 구분하지 못해 헤맸었다. 하지만 이제는 확실히 정리해 헷갈리지 않도록 할 것이다.

undefined 는 var 키워드로 선언한 변수가 암묵적으로 초기화되는 값이다. 즉, 메모리 공간을 할당받고 초기화가 되기 전까지 빈 상태로 있으면 쓰레기 값이 존재하게 되는데, 이러한 값을 남겨두지 않고 undefined 로 초기화 하는 것이다.

이렇듯 undefined 는 개발자가 의도적으로 할당해주기 위한 값이 아니라 자바스크립트 엔진이 변수를 초기화할 때 사용하는 값이기 때문에 개발자가 직접 undefined 를 사용하는 것은 좋지 않다. 그러한 경우에는 null 을 대신 사용하는 것이 좋을 것이다.

DOM 에서 없는 요소를 선택했을 때는 undefined 가 아니라 null 값을 리턴하게 되는데 이 부분은 위 내용과 일관되지 않는 성격을 띄고 있기 때문에 주의해야 한다.

데이터 타입의 필요성

자바스크립트는 Java나 C 처럼 명시적으로 데이터 타입을 지정해주지 않는 동적인 언어이다. 그럼 이러한 동적인 언어인 자바스크립트에서는 어차피 데이터 타입을 자동으로 해석해줄텐데 왜 데이터 타입이라는 개념이 필요할까?

그것은 컴퓨터가 메모리에 자료를 어떻게 저장하고 사용하는지에 대한 내용과 밀접하다. 아무리 자체적으로 변수의 타입을 측정해준다고 한들, 실제로 메모리에 저장해주는 단계에서는 해당 자료(값)을 메모리 어떠한 공간에 얼만큼 사용해야 하는지 또 사용할 때에는 어떻게 해석해야 하는지 등을 알아야 하기 때문이다.

위 코드를 보면 age 라는 변수에 25 라는 숫자 타입의 값을 대입하려고 하고 있다. 이 때 숫자 타입은 8 바이트만큼을 차지하기 때문에 메모리에 공간을 할당하기 위해 8 바이트만큼의 공간을 확보해야 한다는 사실을 알아야한다.그리고 age 라는 변수를 사용할 때 변수가 참조하고 있는 메모리의 주소는 알지만, 한 번에 얼만큼의 바이트를 읽어와야 정보의 손실이 없을지 정확하게 알아야 하고 지금의 경우에는 8 바이트가 된다. 그리고 이렇게 읽는 것 까지 완료했다면 이제 읽어들인 정보를 어떻게 해석할지 알기 위해 데이터 타입이 필요하게 되는 것이다.

정리하자면

  • 값을 저장할 때 확보해야 하는 메모리 공간의 크기를 결정하기 위해
  • 값을 참조할 때 한 번에 읽어 들여야 할 메모리 공간의 크기를 결정하기 위해
  • 메모리에서 읽어 들인 2진수를 어떻게 해석할지 결정하기 위해

데이터 타입이 필요한 것이다.

JavaScript 03 - 표현식(expression) & 문(statement)

작성일 2021-05-11 Edited on 2021-05-12 In JavaScript , Study

표현식(expression)과 문(statement)

용어 정리

표현식(expression)과 문(statement)에 대해서 말하기 전, 몇 가지 용어에 대한 개념을 확실하게 짚고 넘어가보겠다.

값

값이란 표현식(expression)이 평가되어 생성된 결과이다.

이 때 10 + 20 이라는 식이 평가되어 30 이라는 값이 생성된 것이다.

리터럴

리터럴(literal) 이란 사람이 이해할 수 있는 약속된 문자나 기호를 사용해서 값을 생성하는 표기 방식을 의미한다.

위 3 이라는 것은 우리가 아는 아라비아 숫자 3 이지만 자바스크립트 엔진에게는 숫자 리터럴이 되는 것이다. 그리고 ‘hello’는 문자열 리터럴이 되는 것이다. 표현식이나 평가가 없이 그 자체로 값인 것이다.

표현식(expression)이란?

그렇다면 표현식은 무엇일까?

표현식은 값이 될 수 있는 문(statement)이다. 즉 평가되어 값을 생성하거나 기존의 값을 참조하는 것이다.

위는 100 이라는 숫자 리터럴로 변수를 선언하고 초기화한 것이다. 그리고 리터럴은 값으로 평가되기 때문에 리터럴도 표현식이 된다는 점에 주목해야 한다

그리고 위의 예시에서도 123 * 456 이라는 식이 평가되어 값을 가지기 때문에 위의 예시도 표현식에 해당된다.

문(statement)이란?

문은 표현식과 헷갈릴 수 있지만, 이 둘을 잘 구분한다면 자바스크립트 코드를 읽고 실행 결과를 예측하는데 큰 도움이 될 수 있다.

문은 프로그램을 구성하는 기본 단위, 최소 실행 단위이다.

위 한 줄의 코드는 문(statement)이다. 그리고

  • var
  • sum
  • =
  • 1
  • -
  • 2

라는 문법적으로 더 나눌 수 없는 코드의 기본 요소인 토큰들로 이루어진다.

그리고 많이 들어봤던 선언문, 조건문, 반복문 등이 모두 여기서 말하는 “문”에 해당한다.

표현식과 문의 관계

많이 헷갈리는 부분인데, 표현식은 문의 일부일 수도 있으며 그 자체로 문이 될 수도 있다.

이 때 위 코드는 표현식이면서 완전한 문이 되지만, 아래 코드는 변수의 선언문으로써 값으로 평가될 수 없기 때문에 표현식이 될 수 없다.

하지만 “되”와 “돼”를 “하”와 “해”에 대입하여 생각해 구분하는 것과 같이 위 둘도 쉽게 구분할 수 있는 방법이 있다. 그것은 바로 변수에 할당해보는 것이다.

자바스크립트의 변수

변수란?

프로그래밍을 접해본 사람이라면 변수는 “변하는 수” 라고 간단히 알고 있을 것이다. 사실 이는 맞는 말이지만, 조금 더 자세히 보면 더 큰 의미를 가지고 있다.

“변수는 하나의 값을 저장하기 위해 확보한 메모리 공간 또는 그 주소” 라고 말할 수 있다.

위 코드는 정상적으로 동작할까?

물론 동작하지만 의미가 없는 코드이다. 왜냐하면 계산된 50이라는 값을 참조하여 사용할 수 없기 때문이다. 이러한 이유에서 값들에 대한 참조(별명)이 바로 변수가 되는 것이다. 즉, 변수란 어떠한 값을 저장하기 위해 메모리 공간을 확보해 값을 할당하고, 그 공간을 참조(식별)하기 위해 붙여진 이름을 말한다.

메모리 주소를 대신 변수로 접근해야하는 이유

메모리 주소를 잘못 쓰는 등의 실수(human error)가 발생하였을 때 프로그램 작동이 멈추거나 치명적인 오류가 발생할 수 있기 때문이다. 따라서 사람의 실수를 줄이기 위해서 직접적인 메모리 주소 대신 참조가능한 의미 있는 별명인 변수를 사용하는 것이다.

식별자란?

그럼 위에서 말한 식별자란 무엇일까?

위에서 말한 변수 이름을 식별자(identifier)라고도 하는데, 식별자는 어떠한 값을 구별할 수 있는 고유한 이름을 의미한다. 개개인을 고유한 주민등록번호로 식별하는 것과 마찬가지로 메모리에 저장되어 있는 값을 참조하기 위해 그 위치를 구별하고 식별할 수 있어야 한다. 따라서 식별자는 값이 저장되어 있는 메모리의 주소를 기억하고 있는 것이다.

  • 변수
  • 함수
  • 클래스

등은 모두 식별자이다. 그리고 고유해야 한다는 관점에서 코드 내에 무조건 고유해야 한다는 것은 아니고 해당 범위(scope)안에서 유니크해야 한다. 이는 마치 어떤 파일 이름이 해당 폴더 내에서 유니크해야하는 것과 마찬가지이다.

변수 선언

선언이라는 것은 알린다는 뜻이고, 따라서 변수를 선언한다는 것은 자바스크립트 엔진에게 변수를 사용하겠다는 것을 알리는 것이다. 구체적으로는 원하는 값을 저장하기 위한 메모리 공간을 확보하고(allocate) 변수 이름과 그 공간을 연결해 값을 참조할 수 있도록 준비하는 것이다.

변수를 사용하려면 반드시 “선언”이 필요하고, 다음과 같은 키워드를 사용할 수 있다.

  • var
  • let
  • const

ES6 전까지는 var 키워드가 유일한 변수 선언 키워드였다. 하지만 블록 레벨 스코프를 지원하지 않고, 재선언이 가능한 점 등 문제점이 많았다. 그리고 ES6부터 let, const의 등장으로 하위 호환성을 지키며 ES5의 상위호환을 하고 있다.

변수는 선언과 동시에 초기화 할 수 있으며, 만약 선언(var) 후 초기화가 되지 않은 변수는 undefined로 암묵적인 초기화가 수행된다. 그리고 변수 이름을 포함한 모든 식별자는 실행 컨텍스트(execution context)에 등록된다. 이는 코드의 실행 결과를 실제로 관리하는 영역이다.

만약 선언되지 않은 식별자에 접근할 때는 참조 에러(ReferenceError)가 발생하기 때문에 주의해야 한다.

호이스팅

자바스크립트는 다른 언어들과 다르게 매우 신기한 현상이 있었다. 그것은 바로 “호이스팅” 이라는 것이다. 먼저 코드부터 보자.

위 코드는 name 이라는 변수가 선언되기 전에 name 이라는 식별자에 접근하려 했기 때문에 ReferenceError 가 발생하는 것이 당연하는 것처럼 보이지만, 실제로 아주 잘 동작한다. undefined 를 console.log() 로 출력하는 것이다.

다른 언어를 먼저 배웠다면 이는 매우 신기한 현상이 아닐 수 없는데, 이러한 호이스팅이라는 현상이 가능한 이유는 자바스크립트의 엔진의 특성때문이다. 자바스크립트 엔진은 소스코드를 실행하기 위한 준비를 하는 과정에서 런타임 이전에 변수 선언을 포함해서 함수 선언 등의 모든 선언문을 실행한다. 그리고 이렇게 소스코드 실행 준비 과정이 끝나면 처음부터 한 줄씩 다시 제대로 실행되는 것이다. 즉, 변수 선언과 다른 코드의 실행 시점이 다르기 때문에 호이스팅이라는 현상이 발생하는 것이다.

이 때, “선언”만 먼저 수행되는 것이지 “초기화”도 함께 수행되는 것은 아니라는 점에 주의해야 한다.

코드는 일반적으로 위에서부터 아래로 차례로 진행되는 것이 논리적으로 옳고, 헷갈리는 부분이 없다. 호이스팅 덕분에 ReferenceError가 발생하지 않고 동작하게 되는데 이는 예상치 못한 논리적 오류가 에러가 발생할 수 있기 때문에 호이스팅을 이해하고 사용을 지양해야 한다.

변수 네이밍 규칙

식별자는 값을 구별하는 고유한 이름이기 때문에 네이밍 몇 가지 규칙을 준수해야 한다.

  • 식별자는 특수문자를 제외한 문자, 숫자, 언더스코어(_), 달러 기호($)를 포함할 수 있다.

  • 식별자는 특수문자를 제외한 문자, 언더스코어(_), 달러 기호($)로 시작해야 한다. 숫자로 시작하는 것은 허용하지 않는다.

  • 예약어는 식별자로 사용할 수 없다.

키워드는 예약어에 속하는 개념이다.

그리고 네이밍 컨벤션에는 크게 4가지 유형이 존재한다.

  • camelCase
  • snake_case
  • PascalCase
  • typeHungarianCase

자바스크립트에서는 보통 camelCase를 많이 사용하고 생성자 함수, 클래스 등에서는 PascalCase를 사용하는 것이 컨벤션인 것 같아서 앞으로 위 컨벤션대로 개발을 할 예정이다.

네이밍은 개발자에게 있어서 매우 중요하지만 매우 귀찮고 어려운 부분이라고 생각한다. 하지만 네이밍은 코드의 가독성을 매우 높여주어 유지보수의 효율성을 높여주기 때문에 신중하고 의미 있는 네이밍을 해야한다.

자바스크립트와 프로그래밍

자바스크립트는 태생이 애플리케이션 개발용 언어가 아니고, 웹 브라우저를 위한 한정적인 언어이다. 따라서 아주 간단하게 사용되는 경량의, 어떻게 보면 무시받는 언어였다. 또 자바스크립트는 암묵적으로 에러를 처리해주는 기능(magic)이 많아서 장점이라고 볼 수도 있지만 이런 기능이 오히려 개발자들을 힘들게 하는 경우도 많다.

개념과 동작 원리의 중요성

위와 같이 자바스크립트는 태생이 애플리케이션 개발용이 아니었기 때문에 현재 자바스크립트로 개발을 하기 위해서는 내부 동작 원리를 정확하게 알고 사용해야 한다. 왜냐하면 이제는 더 이상 예전의 그 간단하고 쉽게 사용할 수 있는 자바스크립트가 아니고, 제대로 사용하지 않는다면 버그 투성이 애플리케이션이 될 것이기 때문이다.

예를 들어서, 자바스크립트에서는 객체를 생성하는 방법이 총 5개가 있기 때문에 어떠한 상황에서 어떠한 방법을 사용할지 선택해서 적용해야 한다. 이제는 자바스크립트를 깊게 공부해야할 필요성이 생기고 있는 것이다. DOM 컨트롤에서 끝이나는 공부가 아니라 더 깊이, 더 정확한 내용까지 알아야 할 것이다.

공부 방법의 종류

  1. Top-down
  • 흥미를 가지고 공부할 수 있다.
  • 하지만 학습 과정에 있어서 지식에 구멍이 뚫려 아예 들어보지도 못한 부분이 존재할 수도 있다.
  1. Bottom-up
  • 한 번 쭉 훑어볼 수 있어서 지식에 구멍이 뚫릴 일이 없다.
  • 하지만 고된 일이다.

그리고 이번 자바스크립트를 공부할 때에는 아주 탄탄한 기본기, 깊게 공부하기 위해서 Bottom-up 방식을 선택해 아래와 같은 순서로 공부할 것이다.

  1. 기본 개념과 동작원리(이해: 용어 및 개념, 문법)
  2. 코딩 스킬(숙달: 의식적 연습, API, 라이브러리)
  3. 프로젝트(경험)

자바스크립트와 ECMAScript

자바스크립트는 컴파일 언어와 인터프리터 언어 둘 중 하나라고 정확히 단정지을 순 없지만, 따지자면 인터프리터 언어와 더 가깝다고 할 수 있다. 그리고 이러한 자바스크립트는 ECMAScript가 규정하는 핵심 문법을 포함하고 있다. ECMAScript는 자바스크립트의 표준 사양을 말하며 언어의 타입, 객체와 프로퍼티, 함수 등의 핵심 문법을 규정한다. 그리고 이러한 ECMAScript 사양을 준수해 브라우저 내장 자바스크립트 엔진을 구현하게 되는 것이다.

브라우저에서 동작하는 자바스크립틑 DOM이라고 불리는 객체가 있지만 ECMAScript에는 그러한 명세가 없다. 예를 들어, Node.js에서는 다음과 같은 코드가 동작하지 않는다.

JavaScript를 배우는 마음가짐

자바스크립트를 안좋게 보는 사람들이 많이 있다. 내가 생각했을 때 그 이유 중 하나는 javaScript의 탄생 배경이 조금 특별하기 때문이다. 대부분의 프로그래밍 언어는 애플리케이션을 개발하기 위한 범용적인 용도로 설계되었지만, 자바스크립트는 웹페이지의 단순한 보조 기능을 처리하기 위한 제한적인 용도를 목적으로 태어났다. 따라서 자바스크립트를 브라우저 전용 프로그래밍 언어라고 한정지어 생각하는 의견이 없지 않아 있기 때문에 조금 오해를 받는 면이 있는 것 같다.

하지만 현재 자바스크립트는 절대 무시할 수 없을만큼 성장했고, 웹 브라우저에서만 동작하는 것이 아니라 Node.js 라는 자바스크립트 실행환경이 탄생함에 따라 서버, 로보틱스, 모바일, 인공지능 등 그 사용범위가 매우 넓어졌고 앞으로도 계속 넓어질 것이라고 생각한다. 그리고 자바스크립트는 Java나 C와 같이 크게 변동사항이 없이 안정적인 상태를 유지하는 언어들과는 달리, 계속 버전별로 새로운 기능이 추가되어 나가는 진화하는 언어라는 인식이 강하게 박혀있고 성장하는 언어라는 느낌이 너무 강하게 든다.

그래서 개인적으로 문법만 아는 것이 아니라 동작 원리와 개념 등을 심도있게 배우고 싶은 마음이 들기 때문에 자바스크립트라는 언어를 뿌리 깊게 이해하는 것을 목적으로 JavaScript를 공부할 것이다.

나는 javaScript를 “사용할 줄 아는 사람” 이 되는 것이 목표가 아니다. 누가 보더라도 감탄할만한, 잘 모르는 사람이 나의 코드를 보더라도 javaScript에 대한 나의 이해의 깊이가 느껴질만큼 깨끗하고 아름다운 코드를 짤 것이다.

개념과 동작 원리 이해의 중요성

프로그래밍 언어의 기본 개념과 동작 원리를 이해하는 것은 결국 내 생각을 코드로 구현해야하기 때문이라고 생각한다. 그리고 그 이해가 깊으면 깊을수록 더 퀄리티가 높은 코드를 짤 수 있을 것이다. 한번에 모든 것을 학습하려고 욕심내지는 않을 것이다. 그러다보면 놓치는 것이 많아지고 놓친 것을 인지하지조차 못할 수도 있기 때문이다. 또한 학습한 것을 실제로 적용할 때에는 라이브 코딩을 직접 보고만 있는다던가, 이미 구현할 줄 아는 간단한 예제들을 실습하며 나의 실력에 만족하는 의미 없는 연습을 하지 않아야 한다.

연습을 할 때에는, “의식적인 연습” 을 해야만 한다. 의지를 가지고 학습한 것을 녹여낼 수 있으며 나의 실력을 약간 벗어나는 정도의 도전의지를 가지로 임할 수 있는 학습이어야 한다. 앞으로 javaScript를 학습하며 위 내용을 잊지 않고 간직할 것이다.

어떠한 개발자가 좋은 개발자인가

  • “빨리” 보다는 “제대로” 학습하는 사람

  • 코드 한 줄을 짤 때에도, 여러가지 방법들 중 “왜” 하필 그 방법을 선택했는지 설명하고, 책임질 수 있는 사람

  • 되면 왜 되고 안되면 왜 안되는지 정확하게 아는 사람

  • 인터넷 검색을 통해 얻은 결과들을 짜집기 하는 것이 자신의 실력이라고 착각하지 않는 사람

  • 자신의 코드가 어떻게 동작하는지 정확하게 예측하고 코드를 짜는 사람

  • 어떻게든 동작하는 코드를 짜고 만족하지 않고 가독성이 높은, 유지보수가 편한 코드를 짜기 위해 노력하는 사람

  • “왜” 에 대한 물음에 정확하게 답할 수 있는 사람

좋은 개발자가 되기 위해서 화이팅!

빨리 가는 유일한 방법은 제대로 가는 것이다.
- Robert C. Martin, “Clean Code” 저자

프론트엔드 개발자가 하는 일

애플리케이션 사용자가 애플리케이션과 소통하기 위한 창구(UI)를 구현한다.

HTML과 CSS로는 UI 구조를 짜고 예쁘게 꾸밀 수는 있지만, 사용자와 서로 상호작용하는 기능 등을 커버하지 못하기 때문에 JavaScript를 활용하여 커버한다. 이러한 JavaScript를 통해서 잘 꾸며진 화면에서 추가적인 기능이 동작할 수 있도록 한다.

  • 상태정보를 서버로 전송
  • 서버에서 데이터를 받아 UI에 표시

프론트엔드 개발자 필요 덕목

  1. 빠른 학습능력

    프론트는 라이브러리나 툴들이 수시로 변화하기 때문에 빠른 학습능력이 매우 필요하다. 따라서 아주 탄탄한 기본기가 반드시 필요한데, 이 때 javaScript에 대한 깊은 이해가 여기서 말하는 기본기에 해당한다고 생각한다.

    그리고 기본적인 “용어”를 아는 것이 아래 협업과도 큰 관련이 있다. 여기서 말하는 용어란, 개념을 정확히 아는 것이다. 개념을 정확히 아는 것이란, 내가 상대방이 정확히 알 수 있도록 설명할 수 있어야 한다. 함께 일을 하는 과정에서 무엇인가를 설명해야 할 때, 소위 말하는 개떡같이 말해도 찰떡같이 알아듣는 사람과 더 함께 일하고 싶기 마련이다.

  2. 협업 능력

    협업은 대부분 말로, 문서로 하기 때문에 “커뮤니케이션”이 매우 중요하다. 의사소통을 하기 위해서는 서로 알고 있는 지식의 수준이 비슷해야 한다. 위에서 말한 “용어”를 아는 것과도 연관이 있다. 예를 들어 지금의 내가 의사, 변호사들과 해당 분야에 대해서 토론하고 함께 일할 수 없는 것과 마찬가지이다.

    의사소통을 제대로 못하고 무작정 일을 시작하면 서로가 같은 목표를 보고있다고 보장하지 못하기 때문에 아무리 각자가 열심히 일을 한다고 한들 의미가 없이 시간낭비를 하는 것과 마찬가지가 되는 것이다.

비디오 부분 디자인 고려사항

비디오 부분을 디자인할 때, 가장 먼저 눈에 띈 것은 모바일 환경에서 스크롤을 내릴 때 비디오(헤더와 비디오 플레이어 부분)가 화면 상단에 sticy 되는 형상이었다.

Sticky…?

그래서 처음엔 단순히 비디오의 헤더봐 비디오 플레이어 부분에 position: sticky를 주려고 하였다. 하지만 그 결과는 다음과 같았다.

문제가 무엇이었을까… 그것은 바로 position: sticky가 가지는 원래 성질때문이었다. sitcky 속성은 부모 컨테이너 내부에서만 동작하는 성질을 가지고 있었기 때문에 이러한 현상이 발생한 것이다.

So…

마크업을 변경하지 않는 이상 이러한 문제를 해결할 방법을 찾지 못했고, 따라서 javaScript를 사용해 동적으로 is-fixed라는 클래스를 부여해 조건부로 화면 상단에 position: fixed를 해주는 방식을 선택하였다.
위화 같은 선택은 크롬, 파이어폭스, 엣지, 사파리 등 거의 모든 브라우저에서 제대로 동작했지만 역시 우리의 인터넷 익스플로러는 쉽게 동작해주지 않았다.

익스플로러는 미디어쿼리와 동적인 클래스부여가 함께 동작할 때 동적인 클래스의 스타일 속성을 그대로 적용시키지 못하였기 때문이다.

따라서 미디어쿼리를 다음과 같이 변경시켜준 후에야 제대로 동작하게 되었다.

댓글 부분 디자인 고려사항

댓글 디자인은 기술적으로 크게 어렵거나 고려할만한 부분이 없었다고 생각된다. 하지만, 개인적으로 한가지 아쉬운 점이 있었는데 바로 댓글 입력창 애니메이션 부분이었다. 아래는 실제 VLIVE 사이트의 댓글 입력칸인데, 댓글을 입력하기 위해서 클릭할 경우 아래와 같이 한순간에 펼쳐지는 방법을 선택하고 있었다.

하지만 이는 어딘가 부자연스럽고 딱딱한 느낌을 준다고 생각했고, 클론 코딩하는 입장에서 조금 더 사용자의 경험을 좋게하기 위해서 아래와 같이 약간 부드럽게 수정해보았다.

결론

이번 클론 코딩의 주 목적은 접근성와 웹 표준에 목표를 두는 것이었다. 그리고 이번 클론 코딩을 하며 WCAG 2.1 의 기준을 최대한 준수하기 위해 Tab Focus, Color Contrast 및 Logical Sequence 등을 최대한 고려해 약 일주인간 진행해보았다.

그래서 그런지 디자인 상으로는 크게 어려운 점이 없었지만 웹 표준과 접근성을 고려해서 마크업 단계에서부터 Semantic한 태그들의 사용 및 WAI-ARIA를 고려하며 프로젝트를 진행하다보니 작업 속도가 생각보다 더뎠다.

하지만 결과적으로 웹 사이트의 접근성와 SEO 등의 성능을 측정해주는 lighthouse의 측정에서 높은 점수를 획득할 수 있어 뿌듯했고, 실제 VLIVE와 비슷한 수준의 디자인을 뽑아낼 수 있어서 아쉬운 점이 있었지만 그래도 만족스러운 팀 프로젝트였다고 생각한다.
